# ML-IDS — Machine Learning Intrusion Detection System

![Tests](https://github.com/<reallyngb>/ml-ids/actions/workflows/tests.yml/badge.svg)

A complete, modular, end-to-end ML/DL-based Intrusion Detection System:
dataset → preprocessing → EDA → 6 classical ML models → deep-learning
autoencoder → live packet capture → multi-channel alerting → dashboard.

Built as a final-year cybersecurity capstone project.

## How an IDS works

An IDS passively monitors network traffic (via a mirrored switch port or
TAP — never inline) and classifies observations as normal or malicious.
Unlike an IPS, it does not block traffic; it alerts.

**Signature-based IDS** (Snort, Suricata) match traffic against known
attack patterns/rules. Fast and explainable, but blind to zero-days.

**ML-based IDS** (this project) learns statistical patterns from labeled
flow data. Can generalize to novel attack variants, at the cost of needing
good training data and being less directly explainable.

## Architecture

```
Raw Packets/PCAP → Flow Features (CICFlowMeter-style) → Preprocessing →
ML/DL Models → Prediction → Alert System → Dashboard
```

Two pipelines share the same feature schema:
- **Offline**: `data/raw/*.csv` → `preprocessing.py` → `train_ml.py` / `train_dl.py` → saved models
- **Live**: packets → `flow_builder.py` → same scaler/columns → `predict_live.py` → `alert_manager.py`

python3 -m venv venv
source venv/bin/activate        # Windows: venv\Scripts\activate
pip install -r requirements.txt
```

PyTorch is listed in requirements.txt but is large (~1-2GB with CUDA deps).

```

## Dataset

This project targets the **CIC-IDS2017** / **CSE-CIC-IDS2018** flow-feature
schema (the standard academic IDS benchmark datasets from the Canadian
Institute for Cybersecurity, University of New Brunswick).

- CIC-IDS2017: https://www.unb.ca/cic/datasets/ids-2017.html
- CSE-CIC-IDS2018: https://www.unb.ca/cic/datasets/ids-2018.html

Both are multi-GB PCAP + CSV releases covering Benign, DoS, DDoS, PortScan,
Brute Force, Web Attack, Infiltration, and Botnet traffic across 5 days.
CICFlowMeter (Java tool, or `pip install cicflowmeter`) converts raw PCAPs
into the ~80-column flow-feature CSVs the models expect.

**To use the real dataset**: download the "MachineLearningCSV" files,
place them in `data/raw/`, and point `config.yaml -> data.raw_filename`
at the file (or concatenate multiple days' CSVs into one file first).
No code changes are needed — every script only depends on column names.

**Included for immediate use**: `src/data/generate_sample_dataset.py`
generates a synthetic dataset with the identical schema and realistic
per-class statistical separation, so the entire pipeline runs today
without a multi-GB download. Regenerate anytime with:
```bash
python src/data/generate_sample_dataset.py
```

## Running the pipeline

```bash
# 1. Generate/refresh the dataset
python src/data/generate_sample_dataset.py

# 2. Clean, encode, split, scale, balance (SMOTE)
python src/data/preprocessing.py

# 3. Exploratory analysis -> data/processed/eda_plots/
python src/eda/visualize.py

# 4. Train + compare 6 ML models -> models/sklearn/
python src/models/train_ml.py

# 5. Train the autoencoder (needs torch) -> models/pytorch/
python src/models/train_dl.py

# 6. Live detection (needs root/CAP_NET_RAW)
sudo python src/live/predict_live.py

# 7. Dashboard (run alongside step 6 to watch alerts live)
streamlit run dashboard/app.py
```

## Models compared (Phase 5)

| Model | Theory | Strength | Weakness |
|---|---|---|---|
| Decision Tree | if/else splits on thresholds | fast, interpretable | overfits alone |
| Random Forest | bagged ensemble of trees | robust, handles non-linearity | slower, less interpretable |
| XGBoost | gradient-boosted trees | usually best on tabular data | more hyperparameters to tune |
| Logistic Regression | linear boundary | fast, interpretable coefficients | weak on non-linear patterns |
| SVM (RBF) | max-margin + kernel trick | strong on smaller datasets | slow on large data, needs scaling |
| KNN | majority vote of nearest neighbours | simple, no training phase | slow inference, curse of dimensionality |

**Results on the included synthetic dataset** (20k flows, from an actual
`train_ml.py` run — regenerate anytime, numbers will vary slightly run to
run due to random SMOTE/split seeds unless you fix `random_seed` in config.yaml):

| model | precision | recall | f1 | roc_auc | detection_rate | false_positive_rate |
|---|---|---|---|---|---|---|
| decision_tree | 0.934 | 0.965 | 0.949 | 0.984 | 0.965 | 0.023 |
| random_forest | 0.949 | 0.971 | 0.960 | 0.997 | 0.971 | 0.017 |
| **xgboost** | **0.962** | **0.971** | **0.967** | **0.998** | **0.971** | **0.013** |
| logistic_regression | 0.888 | 0.971 | 0.928 | 0.992 | 0.971 | 0.041 |
| svm | 0.912 | 0.984 | 0.947 | 0.995 | 0.984 | 0.032 |
| knn | 0.862 | 0.976 | 0.915 | 0.978 | 0.976 | 0.052 |

XGBoost wins on F1 and false positive rate — consistent with its usual
dominance on structured/tabular data. **These numbers are on the synthetic
dataset only** and should not be cited as real-world performance; re-run
`train_ml.py` against real CIC-IDS2017/2018 data before drawing conclusions
about actual detection capability.

## Why Precision/Recall over Accuracy

IDS traffic is heavily imbalanced (usually >80% benign). A model that
predicts "benign" for everything scores ~80%+ accuracy while catching zero
attacks. **Recall** (= Detection Rate) measures what fraction of real
attacks were caught — missing one matters far more than the accuracy
number suggests. **Precision** measures how many "attack" alerts were
real — low precision buries analysts in false positives until they start
ignoring the IDS entirely. F1 balances both; ROC-AUC summarizes the
tradeoff across all thresholds.

## Zero-day detection (Phase 12)

The autoencoder (Phase 6) is trained ONLY on benign traffic and flags high
reconstruction error as anomalous — it never needs to have seen a specific
attack type to flag it, unlike the supervised models in Phase 5. Combining
both approaches (supervised for known attacks + autoencoder for anomalies)
gives defense in depth.

## Testing

```bash
pip install pytest
pytest tests/ -v
```

17 unit tests cover the parts most likely to silently break and corrupt
results: `preprocessing.py` (cleaning, encoding, stratified splitting,
leakage-free scaling), `flow_builder.py` (bidirectional packet merging,
feature schema matching the trained models), and `alert_manager.py`
(CSV/SQLite writes, ordering, safe no-op on unset webhooks).

## Known limitations

- Live capture computes a reduced ~30-feature subset (see
  `flow_builder.py` docstring) vs. real CICFlowMeter's ~80 features;
  extend `Flow.to_feature_dict()` if you need full parity.
- The synthetic dataset is for pipeline validation, not for publishing
  research results — swap in the real CIC-IDS2017/2018 data before
  drawing conclusions about real-world detection performance.
- `min_seg_size_forward` in live flows is a placeholder (needs TCP options
  parsing to compute exactly).

## Deployment (Phase 13)

- **Docker**: `docker build -t ml-ids . && docker run -p 8501:8501 ml-ids`
  runs the dashboard. For live capture, add `--net=host --cap-add=NET_RAW`.
- **Linux**: grant capture permission without root via
  `sudo setcap cap_net_raw+ep $(which python3)`.
- **Windows**: install Npcap, run as Administrator.
- **Raspberry Pi**: works with the same steps; expect slower SVM/KNN
  training — prefer Decision Tree/XGBoost for on-device retraining.
